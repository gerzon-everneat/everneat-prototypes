Bundle implementation guide

How to build the Surface Cleaner bundle on Shopify Storefront API + your synced CMS. Updated for the current-handle-leads model and the council's findings.

Overview

A soft bundle: the page adds a bottle and a concentrate refill as two real Shopify cart lines in one cartLinesAdd (refill optionally on a subscription selling plan). Not a Shopify "bundle product", no Functions, no Plus.

The model — current handle leads

The bundle is a group of member listings — plastic bottle, glass bottle, refill — with no fixed host. The page works from any member's URL, and the current URL handle is always the primary product:

  • On a bottle handle (surface-cleaner / -glass) → bottle leads, refill is the "Add a refill" companion.
  • On the refill handle → refill leads (refill-first), bottle is the "Add a bottle" companion.

Switching Bottle⇄Refill or Plastic⇄Glass changes the URL handle (router pushState) and re-leads. The lead is presentation only — the same two-line set is added to the cart either way. The order is computed at render: put the member matching the URL handle first.

0 · Prerequisites

Shopify — subscription on the refill

"Subscribe −15%" needs a real selling plan on the refill (native Shopify Subscriptions or your app): 15% adjustment, intervals 4/6/8/12 weeks, tag subscription (the app's getProductSellingPlans.ts queries tag:subscription).

If subscriptions aren't set up yet, ship the bundle one-time only first (skip Subscribe in Phase 3) and gate it behind "a selling plan exists on the refill."

2 · CMS pairing + resolver (member group)

Goal Read the member group and hydrate all members live, ordered so the current handle leads. common/api.ts (existing getFiberyProductByProductId) useFetchDataHook.tsx new: resolveBundle()

No new endpoint — the PDP already fetches fiberyProductQuery; the group rides on it as value.bundle.

async function resolveBundle(bundleDef, currentHandle, userTags) {
  if (!bundleDef?.members?.length) return null          // no group → normal PDP

  // hydrate EVERY member up front → switching lead/material is instant, no refetch
  const members = (await Promise.all(
    bundleDef.members.map(async m => ({ ...m, product: (await getProductByHandle(m.handle)).body?.product }))
  )).filter(m => m.product
              && m.product.variants.edges.some(v => v.node.availableForSale)
              && !shouldExcludeProduct(m.product, userTags))     // drop missing/OOS/hidden

  if (members.length < 2) return null                    // not a viable set for this viewer

  // the current URL handle leads — order it first (no host_handle)
  members.sort((a,b) => (a.handle===currentHandle ? -1 : b.handle===currentHandle ? 1 : 0))
  const lead = members[0]
  return { lead, members,
           bottles: members.filter(m => m.role==="bottle"),
           refill:  members.find(m => m.role==="refill") }
}

Verify entering from each handle (plastic / glass / refill) returns the same group with that handle as lead; an unpublished/OOS member drops out; <2 members → no bundle.

5 · Grouped cart card (deferred — but it's the safety net)

Shows the two lines as one "set" card. The council's caveat: deferring this is what allows partial-bundle desync — most dangerous once the refill is on a subscription (an orphaned recurring refill). So pull it forward to coincide with turning Subscribe on.

  • loadCart.ts add attributes { key value } to cartFields; cart/types.ts add attributes to the line type.
  • ItemShop.tsx group by _addedFromBundleId; switch keys index → node.id; new BundleCartItem; qty/remove on both lines + orphan-repair.

Edge cases & gotchas

  • Missing / unpublished / OOS member → resolver drops it; <2 members → no bundle (never 500 the PDP).
  • Title-based glass/plastic classification (getContainerType) breaks on title drift — warn, don't silently misclassify.
  • Pairing-backend ownership (Supabase vs Fly.io) must be settled — see Source of truth.
  • B2B wholesale (50% code) + subscribe (15%) can stack on an orphan line — model it; don't double-count.
  • Hosted checkout is authoritative — never compute the final price client-side beyond display.
  • No codegen (untyped inline GraphQL) — change the query, the type, and the builder together.

Testing

  1. Per-phase Verify checks above.
  2. Enter the PDP from each handle (plastic / glass / refill) → confirm the right product leads and the companion is the others.
  3. E2E: configure → Add the set → 2 cart lines, refill on subscription, correct totals → checkout holds the discount.
  4. Negative: unpublish the refill → bundle disappears; pick a scent a member lacks → that option disables (no fallback).

Rollout

  1. Ship one-time only if subscriptions aren't configured; enable Subscribe when the selling plan exists.
  2. Gate each bundle with the CMS active flag (instant on/off, no deploy).
  3. Start with Surface Cleaner → validate from all 3 handles → roll the pattern to others by adding CMS member groups.
